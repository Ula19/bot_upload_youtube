## Идеи балансировки прокси/WARP

Чтобы YouTube не заблокировал IP прокси при больших объёмах, можно распределять
запросы между прокси и WARP. Рассмотренные варианты:

### Вариант 1: Round-robin 50/50 ❌
50% юзеров через WARP — медленнее в 4 раза. Плохой UX.

### Вариант 2: Взвешенная 80/20 🟡
80% через прокси, 20% через WARP. Простая, частично разгружает прокси.

### Вариант 3: Балансировка по типу запроса 🟢
- `get_info` (лёгкий API запрос) → 50/50 между прокси и WARP
- `download` (тяжёлая загрузка) → всегда через прокси

**Логика:** YouTube блокирует за частоту API запросов, а не за объём трафика.
Скачивания идут с `googlevideo.com` (CDN) — у него другие лимиты.
Распределяя `get_info` уменьшаем footprint прокси у YouTube API.

### Вариант 4: Размер-зависимая балансировка 🟢
- Маленькие видео (Shorts, < 30 МБ) → WARP
- Большие видео (HD, > 30 МБ) → прокси

**Логика:** разгрузить прокси от мелких запросов, оставить ему критичную HD-загрузку.

### Рекомендуемая комбинация: Вариант 3 + Вариант 4 ✅ ВНЕДРЕНО
- `get_info` → 50/50 (`random.random() < 0.5` в `get_info()`)
- Скачивания < `SMALL_VIDEO_THRESHOLD_MB` (дефолт 30 МБ) → WARP
- Скачивания >= порога → прокси
- Аудио → всегда WARP (m4a обычно маленькое)
- Порог настраивается через `.env`: `SMALL_VIDEO_THRESHOLD_MB=30`
- Логирование в метриках: `routing=hd|small|balanced`

## Известные узкие места

| Компонент | Лимит | Решение |
|-----------|-------|---------|
| Прокси bandwidth | 100 Mbit (~12.5 MB/s) | Апгрейд скорости или несколько прокси |
| Прокси concurrency | 100 threads | OK с большим запасом |
| ThreadPoolExecutor | ~8 потоков | Увеличить через `loop.set_default_executor()` |
| FFmpeg merge | CPU-bound | Больше CPU или менее агрессивное склеивание |
| RAM | 8 ГБ | Достаточно до ~30 параллельных скачиваний |
| `/tmp` место | зависит от диска | Фоновая очистка |
| YouTube IP репутация | ~? запросов/день | Ротация IP |

## Метрики для мониторинга

Грепать в логах:
```bash
docker compose logs bot | grep -E "PRIMARY|METRIC|Fallback"
```

Ключевые показатели:
- `[METRIC] get_info` — должно быть < 5s
- `[METRIC] download_video` `speed=` — должно быть > 2 MB/s
- `[METRIC] upload_video` `speed=` — должно быть > 10 MB/s
- Fallback срабатывает редко (< 5% запросов)
